1 Исходные данные

Краткое описание заказчика: Необходимо создать чат-бот, который сможет консультировать внутренних клиентов Х-Банка по часто задаваемым вопросам.

Задачи, которые нужно решить заказчику:

  1. Автоматизировать часть консультационной работы для снижения нагрузки на центр сопровождения, колл-центр.
  2. Необходимо сократить время ответа на заявки, организовать более быстрое взаимодействии со службой сопровождения.
  3. Должна быть возможность вести онлайн общение со специалистом сопровождения.
  4. Создать систему, при помощи которой можно получать консультации/консультировать не только находясь непосредственно на рабочем месте.
  5. Необходимо просматривать и создавать заявки по различным критериям (доработка, дефект, консультирование).

Проблемы, с которыми он сталкивается при решении задач:

  1. Нет унифицированной системы учёта заявок, поступающих от внутренних клиентов банка.
  2. Большой массив часто задаваемых вопросов, который загружает специалистов сопровождения и снижает эффективность работы. Большая база вопросов и задач имеет ответы. Некоторый вопросы с подсказками или инструкциями сотрудник банка может решать самостоятельно.
  3. Постоянная привязанность к рабочему месту – снижение мобильности и оперативности в устранении проблемы. Текущее решение: Консультация через колл-центр, почту, смежные системы, требующие нахождения на рабочем месте.

2 Ожидания заказчика

Цели:

  1. Внедрить чат-бот между сотрудником сопровождения и it-департаментом банка.
  2. Решить проблему загруженности специалистов сопровождения.
  3. Подготовить структуру часто задаваемых вопросов.
  4. Решить ряд внутренних вопросов службы сопровождения.
  5. Организовать работу с заявками.

Бизнес-требования:

  1. Чат-бот должен позволить пользователям более оперативно решать возникающие проблемы.
  2. Система должна консультировать пользователей по вопросу из категории часто задаваемых.
  3. Система должна перенаправить пользователя на линию консультации при отсутствии вариантов ответа на заданный пользователем вопрос.
  4. В системе должна быть возможность работы с заявками – создавать их, просматривать статус.

4 Ключевые требования, которые удалось собрать из различных источников

4.1 Бизнес-требования

  1. Чат-бот должен позволить пользователям более оперативно решать возникающие проблемы.
  2. Система должна консультировать пользователей по вопросу из категории часто задаваемых.
  3. Система должна предлагать сценарии решения проблемы при отсутствии вариантов ответа на заданный пользователем вопрос.
  4. В системе должна быть возможность работы с заявками – создавать их, просматривать статус.

4.2 Ключевые требования НПА, бизнес-правила

  1. Согласно Федеральному закону от 02.12.1990 N 395-1 (ред. от 26.07.2019) "О банках и банковской деятельности", ст. 26. Банковская тайна система должна хранить, защищать данные пользователей и не передавать их третьим лицам.
  2. Система должна соответствовать закону РФ "Об авторском праве и смежных правах" от 09.07.1993 N 5351-1. Приложение должно быть защищено от копирования.
  3. Система должна соответствовать Федеральному закону "О коммерческой тайне" от 29.07.2004 N 98-ФЗ. Система должна обеспечить безопасность данных, отправляемых в приложении.
  4. Система должна соответствовать Федеральному закону РФ от 27 июля 2006 года № 152-ФЗ «О персональных данных» Система должна обеспечить безопасность данных пользователя.
  5. Система должна ссылаться на обновлённые и действующие нормативные и иные акты Банка.

4.3 Пользовательские требования

  1. Администратор должен иметь возможность управлять базой знаний чат-бота и учётными записями пользователей.
  2. Пользователи со стороны банка-клиента должны иметь возможность получить развёрнутый ответ на интересующий их вопрос в рамках продукта Х-Банк, которым они пользуются, чтобы оперативно решать стандартные ситуации.
  3. Система должна реализовать общение внутренних клиентов Х-Банка с сотрудником сопровождения через чат, чтобы улучшить качество консультаций.
  4. Сотрудники сопровождения должны иметь возможность заниматься только критически важными или нестандартными заявками, чтобы увеличить эффективность работы Сопровождения.

4.4 Системные и нефункциональные требования

4.4.1 Администрирование
  1. Необходимо администрировать систему: обновлять и пополнять базу знаний по «Популярным вопросам».
  2. Администратор должен иметь возможность создавать, редактировать, блокировать и удалять учётные записи.
  3. Администратор должен иметь возможность создавать, редактировать и удалять записи в системе.
  4. Соединение с сервером должно быть защищено протоколами шифрования.
  5. Протокол взаимодействия с Банком- консультантом.
  6. Система должна быть доступна 24/7.
4.4.2 Авторизация
  1. Пользователи должны иметь возможность проходить авторизацию по уникальному логину и паролю для получения доступа к системе.
  2. При возникновении проблем с авторизацией должна быть возможность восстановление доступа через мобильный телефон или электронную почту пользователя.
4.4.3 Аутентификация
  1. Система должна записывать адрес почты пользователей в качестве логинов для пользователей.
  2. Система при первой авторизации должна присвоить случайный пароль, состоящий из 10 символов, который необходимо будет сменить пользователю сразу же после первой авторизации.
  3. В учётной записи пользователя необходимо подтвердить номер мобильного телефона и адрес электронной почты.
4.4.4 Возможность консультирования по популярным вопросам
  1. Система должна консультировать пользователей по вопросу из категории часто задаваемых.
  2. Система должна иметь базу часто задаваемых вопросов.
  3. Система должна отображать пользователю несколько подкатегорий и списки вопросов к любой подкатегории.
  4. Система должна выводить на экран пользователю прописанный ответ на выбранный вариант вопроса из подраздела.
  5. Система должна выводить на экран пользователю готовую инструкцию по решению каждого вопроса, представленного в базе популярных вопросов.
  6. Система должна позволять пользователю сценарий решения вопроса, если его не удовлетворила инструкция по часто задаваемым вопросам.
4.4.5 Возможность взаимодействовать с сотрудником консультации
  1. Система должна иметь функцию запроса звонка с консультантом.
  2. Система должна реализовать общение с консультантом через чат.
4.4.6 Возможность работы с заявками
  1. В разделе «Заявки» должен быть реализован поиск по тегам, поиск по частичному совпадению текста, по номер заявки.
  2. Система должна отображать список заявок в виде плиток.
  3. Система должна по запросу открывать необходимую заявку и предоставлять доступную информацию по ней, а также формировать их в форматах doc, docx, pdf для скачивания.
  4. Система должна дать специалисту сопровождения возможность менять статус заявок.
  5. В системе должна быть реализована возможность вернутся на главный экран с любой точки.
  6. Система должна дублировать создание заявки на почту пользователя.
  7. Должны быть реализованы уведомления для пользователей.
  8. В системе должна быть реализована возможность просмотра запрашиваемой документации и формировании её в форматы doc, docx, pdf для скачивания.

10 Дополнительная информация

Все задачи, поставленные в проекте – MVP. На второй этап можно добавить следующую функциональность:
